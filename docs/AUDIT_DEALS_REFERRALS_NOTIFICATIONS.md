# Аудит Invext: сделки, участие пользователей, реферальная система, уведомления

Только описание текущей структуры и план возможных изменений. Изменения в коде не вносились.

---

## 1. Модели сделок

### 1.1 Таблица `deals` (модель `Deal`)

**Файл:** `backend/src/models/deal.py`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int | PK |
| `number` | int | Порядковый номер сделки (уникальный, для отображения пользователю) |
| `percent` | Decimal(5,2) | Доходность в % (например 3.00) |
| `status` | str(16) | `open` \| `closed` \| `finished` |
| `opened_at` | datetime (tz) | Когда открыта |
| `closed_at` | datetime (tz), nullable | Когда закрыта (сбор) |
| `finished_at` | datetime (tz), nullable | Когда начислена прибыль |

**Связи:** `investments` → список `DealInvestment`.

**Жизненный цикл:**  
`open` → (по расписанию 12:00 UTC+1) → `closed` → (через ≥24 ч джоб раз в 5 мин) → `finished` (при начислении прибыли).

## 2. Модели участия пользователей в сделках

### 2.1 Таблица `deal_investments` (модель `DealInvestment`)

**Файл:** `backend/src/models/deal_investment.py`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int | PK |
| `deal_id` | int | FK → deals.id |
| `user_id` | int | FK → users.id |
| `amount` | Decimal(18,6) | Сумма инвестиции |
| `profit_amount` | Decimal(18,6), nullable | Начисленная прибыль (заполняется при finished) |
| `status` | str(16) | `active` \| `paid` |
| `created_at` | datetime (tz) | Время инвестиции |

**Связи:** `deal` → Deal, `user` → User.

**Логика:**
- Инвестиция создаётся только в открытую сделку (`deal.status == "open"`).
- При создании: списание с баланса через `LedgerTransaction` type `INVEST`, создание `DealInvestment` со статусом `active`, обновление кэша `users.balance_usdt`.
- При переходе сделки в `finished`: по каждой активной инвестиции — возврат тела (ledger `DEPOSIT`) + начисление прибыли (ledger `PROFIT`), `DealInvestment.status = "paid"`, `profit_amount` заполняется.

### 2.2 Где участвуют пользователи

- **Роутер:** `backend/src/api/routers/invest.py` — `POST /api/invest` (body: `user_id` = telegram_id, `amount_usdt`). Вызов `deal_service.invest_into_active_deal`.
- **Сервис:** `backend/src/services/deal_service.py`:
  - `get_active_deal` — текущая открытая сделка;
  - `open_new_deal` / `close_current_open_deal` — открытие/закрытие;
  - `invest_into_active_deal` — создание инвестиции (баланс по ledger, списание INVEST);
  - `accrue_profits_for_due_deals` — начисление прибыли по сделкам, закрытым ≥24 ч назад.

## 3. Текущая реферальная система

### 3.1 Модель (User)

**Файл:** `backend/src/models/user.py`

| Поле | Описание |
|------|----------|
| `ref_code` | Уникальный код (8 символов, генерируется при создании) |
| `referrer_id` | FK → users.id (кто привёл; nullable) |

**Связи:** `referrer` (User), `referrals` (список User).

Один уровень: у пользователя либо нет реферера, либо один (прямой пригласитель). Многоуровневой структуры в БД нет.

### 3.2 Привязка реферера

- **При создании пользователя:** в `user_service.get_or_create_user` передаётся `ref_code_from_start` (из deep link `/start REFCODE`). Если пользователь новый — ищется `User.ref_code == ref_code_from_start`, выставляется `referrer_id`.
- **При существующем пользователе:** если `referrer_id is None` и передан `ref_code_from_start` — реферер привязывается при следующем вызове (логин/обновление через бота).
- **Эндпоинт:** `POST /v1/telegram/auth` (auth.router), body может содержать `ref_code_from_start`. Бот при `/start REFCODE` вызывает auth с этим кодом.

### 3.3 Статистика рефералов

- **Количество рефералов:** в `user_service.get_user_with_stats` считается `count(User.id)` по `User.referrer_id == user.id` → одно число (только 1-й уровень).
- **Оборот команды:** сумма депозитов рефералов (только 1-й уровень):
  - USDT: сумма `LedgerTransaction.amount_usdt` где `type == DEPOSIT` и `user_id in (id рефералов)`;
  - USDC: сумма по `WalletTransaction` (legacy) для рефералов.
- **Отдача в API:** в ответе `/v1/telegram/me` есть `ref_code`, `referrer_id`, `referrals_count`; в боте используются в профиле и разделе «Партнёры».

### 3.4 Бот: партнёрская программа

**Файл:** `bot/src/handlers/partners.py`

- Отображаются уровни 1–5 с процентами (7%, 2%, 1%, 0.5%, 0.5%), но **по факту в API отдаётся только количество рефералов 1-го уровня**; уровни 2–5 в бэкенде не считаются и не хранятся.
- Реферальная ссылка: `https://t.me/BOT?start={ref_code}`.
- Разделы: главный экран партнёров, «Поделиться ссылкой», «Моя команда», «Реферальные бонусы» — везде используются только `referrals_count` и `ref_code`.

**Итог:** реферальная система — один уровень (прямые рефералы), привязка по ref_code из /start, бонусы по уровням в UI заявлены, но в коде не реализованы.

## 4. Текущая логика уведомлений

### 4.1 Источник отправки

Уведомления в Telegram исходят **только с бэкенда** через HTTP-запросы к `https://api.telegram.org/bot{BOT_TOKEN}/sendMessage`. Бот как отдельный процесс сообщения о сделках/выводах не отправляет.

### 4.2 Где и что отправляется

| Событие | Где | Кому | Текст (смысл) |
|--------|-----|------|----------------|
| Сделка открыта | `deal_service._broadcast_new_deal_to_all_users` | Всем пользователям с `telegram_id != null` | «Открыт сбор на сделку #N. Вы можете инвестировать USDT в разделе «💰 Инвестировать»» |
| Сбор по сделке закрыт | `deal_service._broadcast_deal_closed_to_investors` | Только инвесторам этой сделки | «Сбор на сделку #N закрыт, средства идут в работу» |
| Сделка отработана (прибыль начислена) | `deal_service.accrue_profits_for_due_deals` (после начисления) | Инвесторам этой сделки | «Сделка #N отработана. Ваш доход X%» |

### 4.3 Планировщик

**Файл:** `backend/src/main.py` — в `lifespan` поднимается `AsyncIOScheduler`, вызывается `deal_service.init_deal_scheduler(scheduler, async_session_maker)`.

**Джобы в `deal_service.init_deal_scheduler`:**
- **12:00 UTC+1 (каждый день):** закрыть текущую открытую сделку → затем разослать «сбор закрыт» инвесторам.
- **13:00 UTC+1 (каждый день):** открыть новую сделку → затем разослать «открыт сбор» всем пользователям.
- **Каждые 5 минут:** проверить сделки со статусом `closed`, закрытые ≥24 ч назад, начислить прибыль и перевести в `finished` → разослать «сделка отработана» инвесторам.

### 4.4 Чего нет

- Уведомления пользователю о **подтверждении/отклонении вывода** (approve/reject в admin) — не отправляются.
- Уведомления о **успешном пополнении** (после webhook NOWPayments) — не отправляются (можно добавить в обработчик webhook или отдельным сервисом).
- Ошибки доставки Telegram логируются как warning, повторной отправки нет.
- Нет единого слоя «сервис уведомлений»: логика размазана по `deal_service`, вызовы Telegram напрямую через httpx.

## 5. Замечания по согласованности (без изменений)

- **Вывод USDT:** баланс USDT считается по `ledger_transactions` (`get_balance_usdt`). При `approve_withdraw` в `withdraw_service` создаётся только `WalletTransaction` (WITHDRAW COMPLETED). Запись в ledger с типом `WITHDRAW` при approve не создаётся — возможна рассинхронизация баланса USDT, если вывод учитывается только через `WalletTransaction`. Стоит проверить: либо вывод USDT тоже проводить через ledger (добавить LedgerTransaction type WITHDRAW при approve), либо явно документировать, что для USDT используется только ledger и при approve нужно создавать и ledger-запись.

## 6. План возможных изменений

### 6.1 Сделки и участие

- Оставить модели `Deal` и `DealInvestment` как есть, при необходимости:
  - добавить поля (например дата расчётного завершения, тип сделки);
  - вынести константы (минимальная сумма инвестиции 50, расписание) в конфиг.
- Рассмотреть отдельный сервис/репозиторий для «календаря сделок» (когда следующее открытие/закрытие) и отдавать это в API/бот.

### 6.2 Реферальная система

- **Многоуровни:** при необходимости 2–5 уровней — либо считать рефералов рекурсивно по `referrer_id` при запросе, либо ввести таблицу/поля под уровни (например `referral_level`, кэш количества по уровням).
- **Бонусы:** если нужны реальные проценты с депозитов/оборота по уровням — ввести модель начислений (например `referral_bonus`: user_id, referrer_id, level, amount, source_deal_id или source_deposit_id), считать оборот по уровням и при начислении создавать записи в ledger (тип например REFERRAL_BONUS) и обновлять баланс.
- **Единый источник правды:** оборот команды и количество рефералов по уровням считать в одном месте (сервис/метод в user_service или отдельный referral_service), API и бот использовать только его.

### 6.3 Уведомления

- **Единый сервис уведомлений:** вынести отправку в Telegram в отдельный модуль (например `notification_service` или `telegram_notifier`): методы `send_deal_opened`, `send_deal_closed`, `send_deal_finished`, `send_withdraw_approved`, `send_withdraw_rejected`, `send_deposit_credited`. Вызывать их из deal_service, withdraw_service, обработчика webhook платежей.
- **Добавить сценарии:**
  - при approve/reject вывода — отправлять сообщение пользователю (по `user.telegram_id`);
  - при успешном зачислении платежа (NOWPayments webhook) — опционально уведомлять пользователя о пополнении.
- **Надёжность:** очередь повторных попыток при ошибке Telegram (или хотя бы логирование с уровнем error и метрики), не падать из-за сбоя отправки.
- **Конфиг:** возможность отключить рассылки (feature flag или конфиг), чтобы не слать в тестах/на стейджинге.

### 6.4 Выводы (баланс USDT)

- Уточнить учёт вывода USDT: если баланс считается по ledger — при approve вывода создавать `LedgerTransaction(type=WITHDRAW, amount_usdt=...)` и обновлять кэш `users.balance_usdt` (как при INVEST/PROFIT). Либо явно описать текущую схему (WalletTransaction только для истории, списание USDT только через ledger) и проверить все места, где списывается USDT.

---

Итог: структура сделок и участия понятна и согласована с ledger; реферальная система — один уровень, бонусы по уровням в UI не реализованы; уведомления только по сделкам, с бэкенда, без централизованного сервиса и без уведомлений по выводам/пополнениям. План выше даёт направление для доработок без изменения кода в этом аудите.
